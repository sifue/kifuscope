# 画面認識・合法手追跡

## キャリブレーション

`Calibration` は画面左上を原点とした盤面、14種類の持ち駒枚数表示、手番表示の矩形をJSONから読む。盤面は幅・高さをそれぞれ9分割する。端数ピクセルは `numpy.linspace` の境界へ分配するため、全画素がいずれかのマスに含まれる。

盤面が後手視点なら `rotate_board_180=true` とし、画像上のマス順をSFEN順へ180度反転する。解像度、ブラウザ拡大率、棋桜の表示テーマが変わる場合はキャリブレーションとテンプレートを分ける。

## テンプレート

`build-templates` は既知SFENとスクリーンショットから次を自動切り出しする。

- 盤面81マス: 空マス、先後の駒、成駒のラベル別
- 持ち駒: キャリブレーションで指定した各駒種の枚数別
- 手番: 先手・後手別

同じラベルへ複数PNGを保存でき、照合時は全候補のうち平均絶対画素誤差が最小のものを採用する。信頼度は `1 - 平均絶対誤差 / 255` とする。閾値未満の盤面マスは `UNKNOWN` とし、空マスと区別する。

初期局面だけでは成駒、持ち駒1枚以上、後手番のテンプレートが生成されない。既知の追加局面から同じテンプレートディレクトリへ追記する必要がある。

## 局面検証

全要素が確定した観測だけを `BoardState` へ変換し、次を検証する。

- 81マスとSFEN構文
- 双方の王が1枚ずつ存在する
- 盤上の二歩がない
- 盤上と持ち駒を合わせた駒数が上限以内
- cshogiが内部的に正常な局面と判定する

不合格の場合は `recognition_failed` とし、推測SFENを評価しない。

## 合法手追跡

`LegalMoveMatcher` は前回確定局面自身と、cshogiが生成した全合法手を1手指した局面を候補にする。各候補について盤面80%、持ち駒15%、手番5%を基本重みとして観測との一致率を計算する。未確定項目は得点計算から除外する。

最高候補が `legal_match_threshold` 未満、または2位との差が `legal_margin` 未満なら不確定とする。同じ着手後局面が `stable_frames` 回連続した場合だけ現在局面を更新する。これによりアニメーション途中や一時的な誤認識を評価へ流さない。

棋桜の持ち駒欄は駒種・枚数によって表示位置が変わるため、初期運用では盤面81マスを主入力にする。`recognize-image` は持ち駒・手番が未設定でも、盤面が全て確定した場合は `board_observed` を返す。この観測を `StableLegalTracker` へ渡し、初期局面から生成した合法手候補により手番・持ち駒を内部状態として補正する。

状態は次のいずれかになる。

- `ok`: 前局面と同じ、または合法手として安定・確定
- `board_observed`: 盤面81マスは確定したが、単体では完全SFENにできない
- `position_unconfirmed`: 合法手候補はあるが安定フレーム待ち
- `recognition_failed`: 合法候補なし、信頼度不足、候補差不足

`track-images --evaluate` は `ok` かつ未評価の新しいSFENだけをYaneuraOuへ送る。

## 実画像導入時の調整

棋桜の実スクリーンショットについて、盤面の向き、配信解像度、ブラウザ拡大率、持ち駒の表示形式、手番表示を確認する。誤認識率は駒種ごとの混同行列で測定し、閾値を下げる前にテンプレート追加とクロップ修正を優先する。

`docs/sample-screenshot/sample01.png` と `sample02.png` は 2064×1112 の棋桜ウィンドウ画像で、盤面は標準向きで中央固定表示されている。初期キャリブレーションとして `samples/calibration.kiou-2064x1112.example.json` を用意している。

```json
{
  "board": {
    "x": 558,
    "y": 105,
    "width": 928,
    "height": 928
  }
}
```

この値は盤面81マス認識の開始点であり、持ち駒欄と手番欄は未指定である。サンプル画像は正解SFENが未確定のため、テンプレート生成には使わない。テンプレートは必ず、初期局面など正解SFENが分かる画像から生成する。

## Windowsウィンドウキャプチャ

Windowsでは棋桜のウィンドウタイトルが `KIOU` で固定されている前提で、`capture-window` が1フレームをPNGへ保存する。

```bash
uv run python -m kiou_eval capture-window --title KIOU --output captures/kiou.png
```

これはOBSから映像を受け取るのではなく、Kifuscopeが同じウィンドウを直接キャプチャする方式である。ウィンドウが最小化されている場合は取得対象から除外する。タイトルが完全一致しない環境では `--contains` を付ける。
